Motivation

Galaxy deployments on Kubernetes typically run jobs within the cluster using the K8s runner. This limits scalability: cluster nodes are fixed, and large bioinformatics workflows compete for shared resources.

Google Cloud Batch offers an alternative: jobs run on dynamically provisioned VMs with per-tool resource sizing, CVMFS reference data access, and no cluster capacity constraints.

We implemented two runners and benchmarked them on a 12-step variant analysis workflow across three dataset sizes (2GB, 5GB, 10GB).

Key Findings

  • Direct Batch is faster end-to-end (10–14%) due to minimal scheduling overhead (NFS eliminates file transfer)
  • Pulsar computes faster (10–15%) thanks to local SSD I/O, especially for disk-bound tools
  • Batch is 26% cheaper at wallclock rates ($16.86 vs $21.22) — Pulsar's overhead + local SSD erases its compute savings
  • Compute-only costs are equal ($3.52 vs $3.43) — overhead ratio is 4.8x (Batch) vs 6.2x (Pulsar)
  • Both runners auto-size VMs per tool via TPV resource parameters

Future Work

  • GCS object store integration to replace NFS for the direct Batch runner
  • Benchmark additional workflows (RNASeq, ChIP-Seq) at larger scales
  • Spot/preemptible VM support for further cost reduction
  • Hybrid routing: automatically select runner per-tool based on I/O characteristics
